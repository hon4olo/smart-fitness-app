# Backend Architecture 1.0 — API Contract

Versioned REST API contract for future cloud sync and user data management.

## Base rules

- Base path: `/v1`
- Content type: `application/json`
- All timestamps: ISO 8601 UTC strings
- IDs: opaque strings generated by the service or client, depending on entity
- All endpoints return a consistent error envelope
- All endpoints are documented only; no implementation is included here

## Common response envelope

Successful responses should use one of these shapes:

```json
{
  "data": {},
  "meta": {}
}
```

For list endpoints:

```json
{
  "data": [],
  "meta": {
    "page": 1,
    "pageSize": 50,
    "hasMore": false
  }
}
```

For sync endpoints:

```json
{
  "data": {
    "status": "ok"
  },
  "meta": {
    "serverRevision": "rev_123",
    "conflicts": 0
  }
}
```

## Error format

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Human readable summary",
    "details": {},
    "requestId": "req_123",
    "timestamp": "2026-01-02T03:04:05Z"
  }
}
```

### Error fields

- `code`: stable machine-readable code
- `message`: human-readable summary
- `details`: structured diagnostics for clients
- `requestId`: trace identifier
- `timestamp`: server time

## Status code conventions

- `200 OK` — successful read or update
- `201 Created` — new resource created
- `202 Accepted` — async sync accepted for processing
- `204 No Content` — successful delete or logout
- `400 Bad Request` — malformed request or invalid payload
- `401 Unauthorized` — missing or invalid access token
- `403 Forbidden` — authenticated but not allowed
- `404 Not Found` — missing resource
- `409 Conflict` — sync conflict or version mismatch
- `412 Precondition Failed` — stale revision / ETag mismatch
- `413 Payload Too Large` — oversized batch
- `415 Unsupported Media Type` — invalid content type
- `422 Unprocessable Entity` — semantically invalid payload
- `429 Too Many Requests` — rate limited
- `500 Internal Server Error` — unexpected server failure
- `503 Service Unavailable` — maintenance or degraded mode

## Auth

### POST `/v1/auth/login`

Authenticate a user and bind the session to a device.

Request:

```json
{
  "email": "ivan@example.com",
  "password": "secret",
  "deviceId": "device_123",
  "deviceName": "Ivan iPhone",
  "platform": "ios"
}
```

Response `200`:

```json
{
  "data": {
    "accessToken": "jwt_access",
    "refreshToken": "jwt_refresh",
    "expiresIn": 900,
    "user": {
      "id": "user_123"
    },
    "device": {
      "id": "device_123"
    }
  }
}
```

Errors: `400`, `401`, `429`, `500`

---

### POST `/v1/auth/refresh`

Rotate a refresh token and return a new access token.

Request:

```json
{
  "refreshToken": "jwt_refresh",
  "deviceId": "device_123"
}
```

Response `200`:

```json
{
  "data": {
    "accessToken": "jwt_access_new",
    "refreshToken": "jwt_refresh_new",
    "expiresIn": 900
  }
}
```

Errors: `400`, `401`, `429`, `500`

---

### POST `/v1/auth/logout`

Invalidate the active refresh token and device session.

Request:

```json
{
  "refreshToken": "jwt_refresh",
  "deviceId": "device_123"
}
```

Response `204`

Errors: `400`, `401`, `500`

## Sync

### POST `/v1/sync/push`

Push queued local operations to the backend.

Request:

```json
{
  "deviceId": "device_123",
  "baseRevision": "rev_100",
  "operations": []
}
```

Response `200`:

```json
{
  "data": {
    "accepted": 42,
    "rejected": 0,
    "conflicts": []
  },
  "meta": {
    "serverRevision": "rev_101"
  }
}
```

Errors: `400`, `401`, `409`, `412`, `413`, `429`, `500`

---

### POST `/v1/sync/pull`

Pull changes since the last known revision.

Request:

```json
{
  "deviceId": "device_123",
  "sinceRevision": "rev_100",
  "limit": 200
}
```

Response `200`:

```json
{
  "data": {
    "revision": "rev_101",
    "operations": [],
    "snapshotAvailable": false
  }
}
```

Errors: `400`, `401`, `429`, `500`

---

### POST `/v1/sync/conflicts`

Submit conflict decisions or fetch unresolved conflicts, depending on mode.

Request:

```json
{
  "deviceId": "device_123",
  "conflicts": []
}
```

Response `200`:

```json
{
  "data": {
    "resolved": 10,
    "remaining": 2,
    "conflicts": []
  }
}
```

Errors: `400`, `401`, `409`, `422`, `429`, `500`

---

### GET `/v1/sync/status`

Return the current sync state for the authenticated user and device.

Response `200`:

```json
{
  "data": {
    "status": "healthy",
    "lastSyncAt": "2026-01-02T03:04:05Z",
    "pendingOperations": 3,
    "conflicts": 0
  }
}
```

Errors: `401`, `429`, `500`

## User

### GET `/v1/user`

Return the authenticated user profile.

Response `200`:

```json
{
  "data": {
    "id": "user_123",
    "email": "ivan@example.com",
    "name": "Ivan"
  }
}
```

Errors: `401`, `404`, `500`

---

### PATCH `/v1/user`

Update profile fields.

Request:

```json
{
  "name": "Ivan",
  "timezone": "Europe/Berlin"
}
```

Response `200`:

```json
{
  "data": {
    "id": "user_123",
    "name": "Ivan",
    "timezone": "Europe/Berlin"
  }
}
```

Errors: `400`, `401`, `403`, `422`, `500`

## Programs

### GET `/v1/programs`

List training programs.

Response `200`:

```json
{
  "data": []
}
```

Errors: `401`, `500`

---

### POST `/v1/programs`

Create a training program.

Request:

```json
{
  "name": "Upper/Lower 4-day",
  "goal": "Hypertrophy",
  "difficulty": "intermediate"
}
```

Response `201`:

```json
{
  "data": {
    "id": "program_123"
  }
}
```

Errors: `400`, `401`, `422`, `500`

---

### PATCH `/v1/programs/{id}`

Update a program.

Request:

```json
{
  "name": "Upper/Lower 4-day v2"
}
```

Response `200`:

```json
{
  "data": {
    "id": "program_123"
  }
}
```

Errors: `400`, `401`, `403`, `404`, `422`, `500`

---

### DELETE `/v1/programs/{id}`

Soft-delete a program.

Response `204`

Errors: `401`, `403`, `404`, `500`

## Future placeholders

These are reserved for later versions and must not be implemented in this phase.

- `GET /v1/ai/*`
- `GET /v1/community/*`
- `GET /v1/coach/*`

## Notes

- Sync endpoints are device-aware.
- Program data maps to the current `TrainingProgram` domain model.
- The API should treat deletes as tombstones when data must remain syncable.
